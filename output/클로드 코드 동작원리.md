# 클로드 코드 동작 원리

> Claude Code가 어떻게 생각하고, 행동하고, 관리되는지를 정리한 문서.

---

## 1. 에이전틱 루프 (Agentic Loop)

Claude Code는 **컨텍스트 수집 → 행동 실행 → 결과 검증** 3단계를 반복하는 루프로 동작한다.

### 1단계. 컨텍스트 수집
> 파일을 읽고 코드 구조를 파악하여 현재 상황을 이해한다.

- 프로젝트 파일 탐색
- 관련 코드 읽기
- Git 상태 확인
- 에러 메시지 분석

### 2단계. 행동 실행
> 수집한 정보를 바탕으로 실제 작업을 수행한다.

- 파일 수정
- 명령어 실행
- 테스트 실행
- Git 커밋

### 3단계. 결과 검증
> 수행한 작업이 올바른지 확인한다.

- 테스트 통과 여부 확인
- 코드 문법 오류 검사
- 예상 결과와 실제 결과 비교

## 4. 세션과 컨텍스트

### 4.1 핵심 개념

#### 세션 (Session)
- 클로드 코드와의 대화 및 작업 히스토리를 담은 **영구적인 기록 단위**
- 저장 위치: `~/.claude/sessions/` (JSON 형태)
- 터미널 종료나 재부팅 후에도 유지됨
- 프로젝트별로 여러 세션을 가질 수 있음

#### 컨텍스트 (Context)
- 현재 추론을 위해 모델이 실시간으로 참조하는 정보
- 구성 요소: 최근 대화 기록, 읽어온 파일 내용, 터미널 명령 결과값
- **제한 사항:** 컨텍스트 윈도우(토큰 제한) 초과 시 성능 저하 및 비용 상승

---

### 4.2 세션 제어 명령어

| 명령어 | 단축어 | 설명 |
|--------|--------|------|
| `claude` | - | 새로운 세션 시작 (기존 맥락 없음) |
| `claude --continue` | `claude -c` | 마지막 세션 즉시 재개 |
| `claude --resume` | `claude -r` | 세션 목록에서 선택하여 재개 |
| `claude --list-sessions` | - | 모든 세션 ID와 생성 시간 확인 |
| `exit` | - | 현재 세션 저장 후 종료 |

---

### 4.3 Session Fork (세션 포크)

기존 작업 흐름의 **체크포인트**를 생성하여 새로운 독립 세션으로 분리하는 기법.

**사용 시점**
- 현재 코드를 유지한 채 위험한 리팩토링을 시도할 때
- 하나의 프로젝트에서 여러 하위 기능을 동시에 개발할 때
- 컨텍스트가 너무 무거워져 필요한 부분만 떼어내고 싶을 때

**장점**

| 항목 | 내용 |
|------|------|
| 안전성 | 원본 작업 내역 보존 |
| 효율성 | 컨텍스트 오염 방지 및 토큰 최적화 |
| 유연성 | 다양한 시나리오를 동시에 테스트 가능 |

## 5. 컨텍스트 최적화

### 5.1 토큰 관리 — `compact`

대화가 길어지면 컨텍스트가 가득 차 모델이 이전 내용을 잊기 시작한다.

- **`/compact`**: 긴 대화를 요약하여 컨텍스트 공간 확보
- 효과: 비용 절감 + 모델 집중도 향상

### 5.2 파일 수동 관리

- 특정 파일을 컨텍스트에서 제외: `.claudeignore`에 경로 추가
- 불필요한 파일이 너무 많이 읽혔을 때: 새 세션을 열거나 수동으로 정리 지시

## 6. 세션 유지 관리

### 세션 삭제

```bash
# 특정 세션 삭제
claude --delete-session <SESSION_ID>

# 전체 삭제 (로컬 데이터이므로 물리적 삭제 가능)
rm -rf ~/.claude/sessions/*
```

### 세션 활용 전략

| 전략 | 설명 |
|------|------|
| Feature-based Session | 새 기능 개발 / 복잡한 버그 수정 시 새 세션으로 컨텍스트를 깨끗하게 유지 |
| Context-Rich Session | 아키텍처 논의 / 지속적 가이드가 필요할 때 `-c` 옵션으로 맥락 유지 |
